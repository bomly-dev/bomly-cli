# Bomly CLI Specification v1.0

**AI-Executable Implementation Specification**

---

# 1. Overview

## 1.1 Purpose

Bomly is a **dependency intelligence CLI and engine** designed to:

* Explain dependencies
* Analyze supply chain risk
* Verify provenance and attestations
* Generate SBOMs
* Support AI-era development workflows

This document defines:

* CLI commands
* JSON schemas
* execution semantics
* policy system

It is intended for:

* AI agents implementing the system
* engineers building or extending Bomly
* CI/CD integrations

---

## 1.2 Design Principles

1. **CLI is the canonical interface**
2. **All outputs must be machine-readable (JSON-first)**
3. **Deterministic core logic (LLMs optional, never authoritative)**
4. **No duplication of logic across surfaces**
5. **Zero-config by default**
6. **Composable architecture**

---

# 2. System Architecture

```text
Core Engine
    ↓
CLI (bomly)

  - GitHub Action
```

## 3.5 Command: `scan`

### Purpose

Render the dependency graph or generate an SBOM.

### Input

```bash
bomly scan --format <json|text|tree|table>
bomly scan -o <spdx-json|cyclonedx-json>[=<path>]
```

When `--sbom-output` / `-o` is omitted, `scan` renders the dependency graph. `tree` and `table` produce human-readable graph output, while `json` produces structured graph output.

When one or more `-o` values are provided, `scan` emits raw SBOM documents in the requested formats. Repeat `-o` to write multiple SBOM variants in a single invocation.

The execution target is selected with exactly one of:

```bash
--path <filesystem-path>
--url <git-url>
--container <image-reference>
```

### Graph Output JSON

```json
{
  "schema_version": "1.0",
  "command": "scan",
  "project": "ProjectDescriptor",
  "graph": {
    "packages": ["PackageRef"],
    "relationships": [
      {
        "from": "string",
        "to": "string"
      }
    ]
  },
  "metadata": {
    "duration_ms": 0
  }
}
```

### SBOM Output JSON

```json
{ ... raw SPDX or CycloneDX JSON document ... }
```

# 5. Policy System

## 5.1 Config File

Path:

```text
.bomly.yaml
```

---

## 5.2 Schema

```yaml
version: 1

policies:
  fail_on:
    - high_risk
    - vulnerability_high
    - attestation_missing

publishers:
  trusted:
    - repo: github.com/org/repo

attestations:
  require_for_new_dependencies: true
  trusted_workflows:
    - ".github/workflows/release.yml"

packages:
  deny:
    - "pkg:npm/example@*"
```

---

## 5.3 Policy Result Schema

```json
{
  "rule_id": "string",
  "status": "passed | failed",
  "subject": "PackageRef",
  "message": "string"
}
```

# 7. Exit Codes

```text
0  success
1  execution error
2  invalid input
3  resolution failure
4  policy violation
5  verification failure
```

---

# 8. Execution Requirements

## 8.1 Determinism

* Same input MUST produce same JSON output
* LLMs MUST NOT affect core output

---

## 8.2 Performance

* `explain`: <50ms typical
* `diff`: <200ms
* `audit`: <500ms (excluding network)

---

## 8.3 Offline Mode

Core features MUST work without network:

* graph building
* diff
* explain

Network optional for:

* vulnerability lookup
* provenance verification

## 9.3 Output Rules

Every command MUST include:

```json
{
  "schema_version": "1.0",
  "command": "string",
  "metadata": {
    "duration_ms": 0
  }
}
```

### PackageRef

`PackageRef` includes package identity plus structured license data:

```json
{
  "name": "string",
  "version": "string",
  "scope": "string",
  "purl": "string",
  "id": "string",
  "licenses": [
    {
      "value": "string",
      "spdxExpression": "string",
      "type": "string"
    }
  ]
}
```

`licenses` MUST always be present and MAY be empty.
